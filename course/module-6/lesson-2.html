Module 6: AI for Technical Tasks (No Experience Required)

Debugging with AI: Finding and Fixing Problems Faster

Something broke. Maybe you followed the steps in the last lesson and got AI to write you a script, but when you ran it, you got an angry red error message full of technical jargon. Maybe a spreadsheet formula is returning "#REF!" instead of a number. Maybe a web page you built looks fine on your laptop but falls apart on your phone.

Welcome to debugging. It is the part of working with code that frustrates even experienced developers, and it is also the area where AI provides some of its most dramatic value. In this lesson, you will learn how to use AI to diagnose and fix problems quickly, even when you have no idea what the error message means.

What Debugging Actually Is

Debugging is simply the process of finding out why something is not working the way you expected and then fixing it. The term comes from the early days of computing when actual insects would get trapped in computer hardware and cause malfunctions. Today it refers to any kind of problem-solving with code or technical systems.

Here is the important mindset shift: errors are not failures. They are information. Every error message, no matter how cryptic it looks, is the computer telling you exactly what went wrong. The problem is that computers communicate in their own language, and that language is not intuitive to humans. AI bridges that gap by translating error messages into plain English explanations and actionable fixes.

How to Share Error Messages with AI

The single most important debugging skill is learning how to share error information effectively. When something goes wrong, there is usually an error message, a traceback, or some kind of output that describes the problem. How you share that with AI determines whether you get a useful fix or a generic non-answer.

The golden rule: copy the entire error message. Do not paraphrase it. Do not screenshot it and try to describe it. Do not share just the last line. Copy the complete error output and paste it directly into your AI conversation.

Here is what a good debugging prompt looks like:

  1. What you were trying to do: "I am running a Python script that reads a CSV file and calculates monthly totals."
  2. The complete error message: Paste the entire traceback or error output.
  3. The relevant code: If you have it, paste the section of code that is causing the issue.
  4. What you already tried: If you attempted any fixes, mention them so the AI does not suggest the same things.
Pro Tip: When copying error messages, include everything from the first line to the last. The top of an error traceback shows where the problem started propagating, and the bottom shows the specific line where it crashed. Both pieces of information are critical for diagnosis.

Common Debugging Workflows

The "It Does Not Run at All" Scenario

This is the most common situation for beginners. You have code from AI, you try to run it, and it immediately crashes with an error. The fix is almost always one of a few common issues:

  • Missing dependencies: The code uses a library that is not installed on your system. The error usually says something like "ModuleNotFoundError" or "No module named X." The fix is installing the missing library, and AI can tell you the exact command.
  • File path issues: The code expects a file in a specific location and cannot find it. You will see "FileNotFoundError" or similar. The fix is either moving your file or updating the path in the code.
  • Version mismatches: The code was written for a different version of a programming language or library. AI can usually adapt the code to your version once you tell it which version you are using.
  • Syntax errors: A typo, missing parenthesis, or incorrect indentation. These are almost always on the exact line number mentioned in the error.

The "It Runs But Gives Wrong Results" Scenario

This is trickier because there is no error message to work with. The code runs without crashing, but the output is not what you expected. In this case, you need to describe the discrepancy to AI:

"The script runs without errors, but the total in the output file shows $45,000 when I know it should be around $120,000. Here is the code [paste code] and here is a sample of my input data [paste a few rows]."

AI can often spot the logical error immediately. Common causes include filtering out too many rows, using the wrong column, or a mathematical operation that is not doing what you intended. When you describe both what happened and what should have happened, the AI can work backward to find the discrepancy.

The "It Works Sometimes But Not Always" Scenario

Intermittent problems are the hardest to debug because they seem random. But they almost never are. Usually, the code works for most inputs but fails on specific edge cases: an empty cell, a date in an unexpected format, a text value where a number was expected, a special character in a file name.

When this happens, share with AI: the code, an example of an input that works, and an example of an input that fails. The difference between those two inputs usually reveals the bug immediately.

Reading AI Explanations of Code

When AI fixes a bug, it usually explains what was wrong and what it changed. Learning to read these explanations, even if you cannot read the code itself, is a valuable skill.

Pay attention to phrases like:

  • "The issue was that the code was comparing a string to a number" (a type mismatch)
  • "The loop was starting at index 1 instead of index 0, skipping the first row" (an off-by-one error)
  • "The function was overwriting the variable on each iteration instead of accumulating" (a logic error)
  • "The date parser expected YYYY-MM-DD format but the data uses MM/DD/YYYY" (a format mismatch)

You do not need to understand the code syntax to understand these explanations. They describe logical problems in plain English. Over time, reading these explanations will give you an increasingly useful intuition for what kinds of things go wrong with code, which makes you better at preventing problems in the first place.

Pro Tip: After AI fixes a bug, ask it "Can you explain in simple terms what went wrong and why this fix works?" This builds your technical intuition over time without requiring you to learn programming formally.

The Debugging Conversation

Effective debugging with AI is rarely a single question and answer. It is a conversation. Here is what a typical debugging session looks like:

  1. You: Share the error message and context.
  2. AI: Identifies the likely cause and suggests a fix.
  3. You: Apply the fix and report what happened (either it worked, it produced a new error, or it changed the behavior but not correctly).
  4. AI: Adjusts its approach based on your feedback.
  5. Repeat until the problem is resolved.

This iterative process is exactly how professional developers debug, except they are having the conversation with themselves, Stack Overflow, or a colleague. You are having it with AI, which is often faster because AI has immediate access to knowledge about virtually every programming language, library, and common error pattern.

When AI Debugging Helps vs When to Call a Human

AI is extraordinarily good at debugging certain categories of problems and limited in others. Understanding the boundary will save you time and frustration.

AI excels at:

  • Syntax errors and typos in any programming language
  • Common runtime errors with well-known causes
  • Library usage mistakes (wrong function name, wrong parameter order)
  • Data format mismatches and type conversion issues
  • Explaining what existing code does and why it might be failing
  • Suggesting alternative approaches when one method is not working

Call a human when:

  • The problem involves your specific network, server, or infrastructure configuration
  • The issue is related to permissions, security settings, or organizational IT policies
  • You have gone through three rounds of AI-suggested fixes and the problem persists (this usually means the issue is environmental, not in the code itself)
  • The code involves sensitive data and you are not comfortable sharing it with an AI service
  • The bug has real financial consequences and you need absolute certainty about the fix
Pro Tip: If AI suggests a fix and it does not work, do not just say "that did not work." Tell the AI exactly what happened: the new error message, the unexpected output, or the changed behavior. The more specific your feedback, the faster you converge on the solution.

Building a Debugging Mindset

The most important thing to internalize about debugging is that it is normal. Professional developers spend a significant portion of their time debugging. Code rarely works perfectly on the first try. Errors are not a sign that you did something wrong. They are a natural part of the process.

With AI as your debugging partner, you have something that professional developers did not have until recently: an infinitely patient assistant that has seen virtually every error pattern before and can explain problems in whatever level of detail you need. Use that advantage.
